# Review 报告：feat/p0-prototype（P0 原型实现）

> 2026-07-17 · 审查对象：`feat/p0-prototype` HEAD = `823fb5e`（GLM + OpenCode 产出的 12 个 commit）
> 方法：6 视角并行审查（Claude workflow 2 个 + opencode/GLM worker 5 个，飞书 WS 协议视角复用前者成果）→ 运行时探针逐字复现 → 关键出入由 Claude 对照 `.venv` 已装 SDK 源码终审。
> 证据等级标注：【复现】= 探针逐字输出；【源码】= SDK/官方参考实现 file:line；【推断】= 机制清晰但未运行时复现。

## 结论摘要

P0 原型的**骨架方向正确**（ACP 封装、自实现 WS 长连接、Defender 兼容 shim、节流器都成立），但**两条 P0 验证声明在 HEAD 上均不可复现**：agent 输出→飞书链路因 `await` 同步方法在每个 chunk 上抛 TypeError 而全断（异常被 ACP SDK `contextlib.suppress` 静默吞掉，所以表面「能跑」）；话题双向通信因 agent 在首轮结束即被关闭而必然拒绝后续回复。`daemon.py` 全部内容来自单一 commit `b1e7926` 且从未修复——声明的「E2E 验证通过」无法在任何已提交版本上成立。共确认 **4 个 critical、11 个 major、7 个 minor**；6 个候选被证伪。修复面集中在 `daemon.py` 的生命周期设计与 4 个点状 bug，工作量可控。

---

## CONFIRMED — critical

### R1. agent 输出链路全断：`await throttler.feed(text)` 对同步方法 await
- 位置：`daemon.py:147`（根因 `throttler.py:42`，`feed` 是 `def`）
- 证据：【复现】`TypeError: object NoneType can't be used in 'await' expression`；【源码】acp `connection.py:268-270` `_run_notification` 用 `contextlib.suppress(Exception)` 包住 handler——异常被静默吞掉，agent 继续跑但**零输出进入节流器**。飞书话题只能看到 daemon 自己的三条状态文案。
- 修复：`on_output` 改为同步调用 `throttler.feed(text)`（feed 本就是为回调设计的同步方法）。

### R2. 话题回复对同一 `AcpAgent` 二次 `start()`：进程泄漏 + 上下文丢失
- 位置：`daemon.py:211-214` → `daemon.py:176` → `acp_client.py:163-199`
- 证据：【源码+推断】`_run_agent_turn` 首行无条件 `agent.start()`；`start()` 不判 `_closed`/已启动，直接重跑 spawn/initialize/new_session 并覆盖 `_conn/_session_id/_transport_ctx/_proc`。首个进程成孤儿（永不被 `__aexit__`）。注释「复用同一 session（保留上下文）」与实现相反。两条回复并发到达时对共享对象的写入无锁竞争。
- 修复：见 R3（同一生命周期设计问题）。

### R3. 首轮 `finally` 即 `aclose()` agent，「可在话题内继续回复」是假的
- 位置：`daemon.py:191-193`（关闭）vs `daemon.py:187`（提示可继续）vs `daemon.py:204-209`（回复被拒）
- 证据：【源码+推断】首轮结束 → `task.done()` 为真 → 一切话题回复命中「⚠️ 该 agent 已结束本轮」。与 CLAUDE.md P0 验证 #2 直接矛盾。
- 修复（R2+R3 一体）：重新设计 turn 生命周期——agent 进程/session 跨 turn 存活；每个 agent 一个 prompt 队列 + 单消费者 task 串行执行 turn；话题回复只入队（session 天然保留上下文）；显式 `/stop` 或空闲超时才 `aclose`。

### R4. 权限请求 100% 失败：`RequestPermissionResponse(outcome="allow")` 非法
- 位置：`acp_client.py:64-68`
- 证据：【复现】pydantic `ValidationError`（`outcome` 是 `DeniedOutcome | AllowedOutcome` discriminated union，合法判别值是 `"selected"`/`"cancelled"`）；【源码】acp `connection.py:251-256` 捕获 ValidationError → 给 agent 回 `-32602 invalid_params` JSON-RPC error。CLAUDE.md「部分文件操作未落盘」的根因即此——不是 Copilot 内部策略，是 client 每次都回错误。
- 修复：`request_permission` 签名改为 `(self, session_id, tool_call, options, **kwargs)`，从 `options` 里选 `allow_once`/`allow_always` 的 `option_id`，返回 `RequestPermissionResponse(outcome=AllowedOutcome(outcome="selected", option_id=...))`。

## CONFIRMED — major

### R5. WS ACK 用 `{"code": 0}`，官方是 `code: 200` → 「消息偶发重复」的头号嫌疑
- 位置：`feishu.py:214-224`；【源码】官方 `lark_oapi/ws/client.py:337`（`Response(code=http.HTTPStatus.OK)`，即 200）+ `:348`（附 `HEADER_BIZ_RT` 耗时头）。服务端判 ACK 无效 → 重推事件 → 重复 spawn。CLAUDE.md 猜测的「WS ACK 回环」无机制支撑（ACK 是出站帧，不会回流进 `_handle_frame`）。
- 修复：ACK payload 改 `{"code": 200}`（错误路径保持 500 与官方一致），补 `biz_rt` header；另在 daemon 层按 `message_id` 幂等去重兜底。

### R6. ping 永不成功：`ws.response.request.url` 属性不存在
- 位置：`feishu.py:175`
- 证据：【复现】websockets 15.0.1 的 `Response`（`http11.Response` dataclass）字段只有 `status_code/reason_phrase/headers/body`，无 `.request`/`.url`。AttributeError 每 120s 被 `except` 吞一次并 debug 日志——ping task 不死但**一个 ping 都发不出**，保活靠断线重连兜底。
- 修复：在 `_ws_connect_once` 里把 `conn_url` 作为参数传给 `_ping_loop`（连接 URL 本来就在手上）；ping 间隔与 service_id 参照官方实现。

### R7. 多分片事件永远组装不出 + 合包缓存三重缺陷
- 位置：`feishu.py:226-242`
- 证据：【复现】三个分片全到达，组装结果 None。机制：(a) `call_soon_threadsafe(pop)` 是**立即**调度而非注释声称的 5s TTL，首分片入缓存即被清；(b) 占位用空串 `""`，`all(buf)` 对空串恒 False，逻辑本身也错（应 `[None]*total` + `is not None` 判定）；(c) `_combine_cache` 是**类属性**，跨实例共享 + WS 线程/主 loop 双线程无锁读写。
- 修复：实例属性 + `None` 占位 + `time.monotonic()` 真 TTL + 单线程内处理（无需跨线程 pop）。

### R8. 子进程 stderr PIPE 从不消费：缓冲区满则 agent 整体卡死
- 位置：`acp_client.py:181`（`spawn_stdio_transport` 默认 `stderr=PIPE`，【源码】`transports.py:53`；全文无人读 `proc.stderr`）
- 影响：stderr 写满 OS 管道缓冲后 agent 阻塞，`prompt()` 永不返回；且 agent 崩溃诊断信息全部丢失。
- 修复：后台 task 持续 drain `proc.stderr` 进日志。

### R9. 完整 `os.environ` 透传 agent 子进程，击穿 SDK 白名单防护
- 位置：`acp_client.py:171-172`；【源码】`transports.py:33-44` 的 `default_environment()` 本只继承 12 个白名单变量，被整份 `os.environ` merge 覆盖。本机全部机密（API key、token）暴露给 agent，而 agent 输出又全量转发到飞书。
- 修复：只传 `self._spawn.env`，让 SDK 自行合并白名单。

### R10. `chat_id=""` 默认全放行 + 无发送者鉴权
- 位置：`config.py:53`、`daemon.py:83`、`config.example.toml:8-10`
- 影响：机器人所在任意群的任意成员可 `/run` —— 等价于把本地 shell 半开放给飞书。example 还把 footgun 当引导。
- 修复：`chat_id` 必填（空则拒绝启动，发现模式单独做成 `--discover` 标志）；加 `sender_whitelist` 配置。

### R11. `/run` 无并发上限：进程炸弹
- 位置：`daemon.py:114-170`。与 R5 的消息重推、R2 的二次 spawn 叠加放大。
- 修复：`MAX_CONCURRENT_AGENTS`（建议 3）+ 同根消息去重。

### R12. followup task 无人持有：异常静默 + `_shutdown` 看不见
- 位置：`daemon.py:211-214`（对比首轮 `:167-170` 有保存）。`task_obj.task` 不更新，Ctrl-C 时 followup 既不被 cancel 也不被 await。
- 修复：并入 R3 的队列设计后自然消失。

### R13. WS 线程死亡 daemon 无感知（静默失联）
- 位置：`daemon.py:72`（主 loop 纯保活）、`feishu.py:124-129`（线程异常仅日志）
- 影响：对调度器而言静默失联比崩溃更糟——用户以为任务在跑。
- 修复：主 loop 周期检查 `_ws_thread.is_alive()` / `last_msg_ts` 看门狗。

### R14. 飞书 HTTP 无重试无退避：限流即丢消息
- 位置：`feishu.py:161/309/325`（裸 `requests.post`）。agent 突发输出 → 节流批次连发 → 触飞书 QPS 限流 → `_safe_reply` 吞异常 → 该批输出永久丢失。
- 修复：`requests.Session` + `Retry(total=3, backoff_factor=0.5, status_forcelist=[429,500,502,503])`，429 读 `Retry-After`。

### R15. 日志/异常泄密面
- 位置：`feishu.py:218`（错误时打印 payload 含用户消息原文）、`daemon.py:190`（异常原文发回飞书群）、`acp_client.py:174`（argv 可能含 token）、`feishu.py:334`（完整 API 响应入异常）。
- 修复：payload 只记 message_id；发飞书的异常文本通用化；argv 打码敏感参数。
